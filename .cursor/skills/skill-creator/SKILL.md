# 技能创建器（Skill Creator）

## 核心定位

通过结构化访谈收集需求，验证该能力不是 Claude 内置功能，设计触发描述和工作流，从失败模式中提炼 Gotchas，生成符合规范的 SKILL.md 文件，并通过质量自检确保文件可用。

---

## 触发场景

- 用户说"帮我创建一个新技能"或"帮我写一个 SKILL.md"
- 用户描述了重复性工作流程，希望封装为可复用单元
- 用户想优化现有 SKILL.md 的触发描述或工作流
- 用户说"我想让 Claude 记住怎么做 X"

---

## 工作流

### Step 1：需求访谈

**按顺序提问，每问等待回答后再继续。不要一次性列出所有问题。**

```
必问（按顺序）：

Q1：这个技能要解决什么问题？
    期望：用一两句话描述核心痛点

Q2：什么情况下会触发这个技能？
    追问：请给我 2-3 个具体的触发例子。

Q3：这个技能的输出是什么？
    追问：能给我看一个理想输出的例子吗？

Q4：这个技能需要哪些外部工具或服务？
    期望：浏览器、API、本地命令、数据库等

Q5：这个技能最容易在哪里出错？你之前手动做这件事时遇到过什么坑？
```

根据回答补充追问（非必问）：
```
Q6：需要用户提供哪些输入参数？
Q7：有没有明确不应该触发的场景？
Q8：执行过程中是否需要用户中途确认？
```

### Step 2：验证非内置能力

在设计技能前，判断用户描述的能力是否已是 Claude 内置能力：

**这些不需要创建技能（内置功能）：**
- 写代码、解释代码、debug
- 翻译、摘要、改写、润色文本
- 数学计算和推导
- 格式转换（JSON/Markdown/CSV/YAML）
- 回答知识性问题

**需要技能的场景：**
- 访问特定外部服务（需要认证的 API、特定网站）
- 执行特定工具链（多步命令行操作组合）
- 强制遵循特定输出模板（标准化报告格式）
- 处理平台特有内容格式（XHS 截断、Notion 块结构）
- 组合多个工具的复杂工作流

如判断为内置能力，告知用户：
```
这个能力是 Claude 的内置功能，不需要专门的技能。
直接说"{具体指令示例}"即可使用。

如果你有特殊要求（固定输出格式、特定处理流程），告诉我，
我们可以创建针对你需求的专属版本。
```

### Step 3：设计技能元素

**名称设计规则：**
- 格式：`lowercase-hyphen`，2-4 个单词，无空格无大写
- 反映动作而非主题：`analyze-sentiment` 而非 `sentiment`
- 示例：`github-pr-reviewer`, `meeting-notes-formatter`, `sql-query-builder`

**触发描述（description）设计规则：**
- 50-150 个字符（中英文混合时按字符计，不按字数）
- 第一个词是动词：Analyzes / Creates / Extracts / Converts / Generates
- 包含触发条件："when user wants to..." 或 "当用户需要..."
- 测试方法：仅凭描述，能否判断何时该触发、何时不该触发？

**工作流设计规则：**
- 步骤数 3-8 个（更少意味着不够具体；更多则应拆分为子技能）
- 每步说明：做什么、怎么做（具体工具/方法）、遇到什么情况怎么处理
- 必须包含错误路径，不只有 happy path
- 每步输入输出明确，不出现"处理数据"这种模糊描述

**Gotchas 来源（从多角度挖掘）：**
- 用户访谈 Q5 中提到的失败经历
- 外部服务的已知限制（限流、认证、格式）
- 边界条件（空输入、超大输入、特殊字符）
- 与其他技能的依赖/冲突

### Step 4：生成 SKILL.md

使用以下模板：

```markdown
---
name: {name}
description: {description，50-150字符}
---

# {中文标题}（{English Title}）

## 核心定位

{2-4 句话：核心价值、适用范围、不适用范围}

---

## 触发场景

- {触发条件 1}
- {触发条件 2}
- {触发条件 3}

---

## 工作流

### Step 1：{步骤名}

{详细说明，含具体操作方法和错误处理}

### Step 2：{步骤名}

{详细说明}

（3-8 步）

---

## 输出格式

{具体格式，含示例，不只是描述}

---

## Gotchas

{至少 4 条，每条具体可操作，不是泛泛的警告}
```

### Step 5：质量自检

生成后逐项检查，全部通过才能输出文件：

```
质量检查清单（13 项）：

格式检查：
[ ] name：lowercase-hyphen，无空格无大写无中文
[ ] description：50-150 字符范围内
[ ] description：以动词开头
[ ] description：包含触发条件（何时/为何触发）

结构检查：
[ ] 五个必要章节全部存在：核心定位/触发场景/工作流/输出格式/Gotchas
[ ] 工作流步骤数 3-8 个
[ ] 总行数 ≤ 500 行

内容质量检查：
[ ] 每个工作流步骤有具体操作说明（非模糊描述）
[ ] 工作流包含至少一个错误处理路径
[ ] 输出格式含具体示例（非仅描述）
[ ] Gotchas 至少 4 条，每条具体可操作

完整性检查：
[ ] 无未填充的占位符（{待填写} 等）
[ ] 无 README 风格的介绍段落（"这个技能是为了..."）
```

未通过任何一项，修正后重新检查，直到全部通过。

### Step 6：确认路径并写入

展示完整 SKILL.md 内容，说明写入路径，等待用户确认后写入。

默认路径推断逻辑：
- 浏览器相关 → `04-浏览器自动化/{技能名}/`
- 开发工具类 → `03-开发工具/{技能名}/`
- 其他类别 → 询问用户应放入哪个分类目录

---

## 输出格式

```
## 技能创建完成

名称：{name}
写入路径：{绝对路径}/SKILL.md
描述：{description}（{字符数} 字符）

质量检查：通过 {n}/13 项
{如有未通过项，列出并说明已修复或需要用户补充}

---

{SKILL.md 完整内容预览}

---

文件已写入。
```

---

## Gotchas

**用户描述太抽象时不要开始设计**
- "帮我自动化工作流" → 追问：具体是哪个步骤最烦？
- "处理数据" → 追问：什么格式的数据？处理完变成什么？
- 基于模糊描述设计出的技能触发条件也会模糊，后续无法正常工作

**description 字符数陷阱**
- 中英文混合时，一个中文字符 = 一个字符（不是两个）
- 用 `len("description text")` 验证，不要目测

**触发条件设计的两个极端**
- 过于宽泛：`description: Helps with coding` → 几乎所有任务都触发
- 过于具体：`description: Only for CSV files with semicolons` → 几乎不触发
- 目标：能正确区分"该触发"和"不该触发"的典型场景

**名称冲突处理**
- 写入前检查目标目录是否已有 SKILL.md
- 如已存在，询问用户是创建新版本（新目录）还是覆盖现有版本

**不适合封装为技能的情况**
- 只用一次的一次性任务（技能是给重复场景用的）
- 需要大量上下文才能执行的任务（技能应当自包含）
- 实际上是多个独立技能的组合 → 拆分后分别创建
- 整个"工作流"只有一步 → 直接作为 Claude 指令使用，不需要技能
