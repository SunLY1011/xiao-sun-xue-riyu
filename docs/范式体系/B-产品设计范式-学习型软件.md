## 一、发心 · 四条原点（改进为了什么）

| 原点 | 目的 | 改进时对照 |
|------|------|------------|
| **学习优先** | 学员时间花在可验证练习上 | 新功能是否增加**有效练习次数**？ |
| **课本同构** | 降低认知切换成本，跟课堂走 | 首页/目录是否仍对齐**教材课序**？ |
| **共同称谓·体验官** | 产品决策可追溯、可讨论 | 是否未经拍板就改 IA？ |
| **100% 才给星** | 诚实反馈掌握度，不刷成就感 | 星级是否被「点过」稀释？ |

**趣味与奖励的定位**：插画、彩蛋、漫画收藏 = **通关后的奖励**，不替代四关；グルミ/绘本叙事服务记忆，不替代课文。

## 二、信息架构三层（固定骨架）

```text
层1  目录/地图     →  我在哪、还剩什么（路径、单元、00入門）
层2  课内关卡      →  我怎么学（本案：単語·会話·文法·テスト）
层3  督导/笔记     →  我学得怎样（进度、薄弱、关联、自笔记）
```

**禁止**：8 课平铺成「全书已上线」超市；**推荐**：地图/路径 + 分区 + 课序节点。

**入門**：L1 的隐藏关 — 不锁正课、可随时查发音（本案 Part-0 / intro.html）。

## 三、种子 · 冻结 · 分波（内容策略）

| 阶段 | 做法 | 目的 |
|------|------|------|
| **种子** | 1–3 课四关完整跑通 | 验证 IA + 数据形状 + TTS 链 |
| **金标** | 定参照课（本案 L14） | Review 有对照尺 |
| **冻结** | tag + 课内不改 | 内测可发、回归有锚 |
| **分波** | Wave A/B/C/D · 金/银/铜 | 避免 24 课同深度一次性幻觉 |
| **解锁** | 用户明示 + baseline 改 status | 内容层与五关壳分开管 |

## 五、设计原则 · 四层拆解（PM 导演手册）

收到任何需求，按顺序拆：

```text
用户表述
  ↓ 教学逻辑   「先音表地图，再分块，再开口」
  ↓ 产品逻辑   「入門=全景三层；首页=路径；课内=四关」
  ↓ 界面逻辑   「一层导航一件事；日语主、中文辅」
  ↓ 实现边界   「动 catalog/journey；不动冻结课内」
```

真源：`docs/PRODUCT-PM-全局导演手册.md` §2

## 六、UI 美化原则（本案拍板 · 可迁移）

| 原则 | 要求 | 为什么 |
|------|------|--------|
| **平面轻量** | 无重阴影/渐变；块用色板令牌 | 微信 WebView 性能与可读 |
| **高度预算** | `100dvh − 顶栏 − 底栏` | 首屏不挤、底栏不漂 |
| **图例即语义** | L0 线性 SVG：书/人/あ | 少文字多图形 |
| **手风琴单开** | 单元/课次同时只展开一个 | 移动端扫视 |
| **日语优先** | 折叠主标题日文；中文辅 | 学习场景 |
| **双界面一致** | 390×844 壳 ≡ 微信同套 H5 | 禁止桌面全宽冒充验收 |
| **全景 Soul** | 先领域池再控件；禁止小李漫画进条带 | 叙事一致（グルミ） |

真源：`DESIGN-L0-图标与人设全景.md` · `学员端双界面显示标准-首要执行.md` · 汇编 §0 R1–R13

## 八、新 App 产品检查清单

1. 是否有**课序/能力表**唯一真源？  
2. 首页是否是**路径/地图**？  
3. 是否有**种子课四关**再批量？  
4. 督导层能否看到**进度与薄弱**？  
5. 趣味资产是否**机制先行、美术分期**？  
6. 发版是否有 **cache + 硬闸门**？  
7. 是否有 **讨论优先 + iteration-baseline**？  
8. 收官是否有 **报告 + 手机清单 + tag**？
